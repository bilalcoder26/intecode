Explain single-threaded architecture in Node.js

## Single-Threaded Architecture in Node.js

The **single-threaded architecture** means Node.js uses **only one thread** to execute JavaScript code, 
unlike traditional servers that create a new thread for each request.

---

## The Core Concept

### Traditional Multi-Threaded Server (Apache, Java Servlets)
Request 1 → Thread 1 (1MB stack) → [====waiting====] → Response
Request 2 → Thread 2 (1MB stack) → [==waiting==] → Response
Request 3 → Thread 3 (1MB stack) → [========waiting========] → Response
Request 4 → Thread 4 (1MB stack) → [waiting] → Response
Request 1000 → Thread 1000 (1MB stack) → [===waiting===] → Response

Memory: 1000 threads × 1MB = 1GB minimum overhead
CPU: Context switching overhead between threads

### Node.js Single-Threaded Server

Single Thread (JavaScript execution)
    ↓
Request 1 → Start → "I'll wait async" → Continue
Request 2 → Start → "I'll wait async" → Continue  
Request 3 → Start → "I'll wait async" → Continue
Request 1000 → Start → "I'll wait async" → Continue
    ↓
When I/O completes, callbacks execute on same thread

Memory: ~10MB (not per-request)
CPU: No context switching overhead

---

## How Single Thread Handles Concurrent Requests

The **magic** is that JavaScript execution is single-threaded, but **waiting happens outside** the main thread.

const http = require('http');
const fs = require('fs');

const server = http.createServer((req, res) => {
    if (req.url === '/database') {
        // This is NON-BLOCKING
        database.query('SELECT * FROM users', (err, rows) => {
            // Callback runs LATER when data arrives
            res.json(rows);
        });
        
        // Main thread immediately returns to event loop
        // Can handle the next request RIGHT NOW
    }
});

server.listen(3000);

### Visual Timeline

Time ─────────────────────────────────────────────────►

Thread 1: [Req1 start][Req2 start][Req3 start]...[Callback1][Callback2][Callback3]
          └─┬─┘     └─┬─┘     └─┬─┘                └─┬─┘      └─┬─┘      └─┬─┘
            │         │         │                   │         │         │
        Req1 I/O   Req2 I/O   Req3 I/O           Req1 done  Req2 done  Req3 done
        (waiting)  (waiting)  (waiting)
```

**Notice:** One thread handles all requests and all responses!

---

## What Does "Single-Threaded" Actually Mean?

### The JavaScript Thread (Main Thread)
- Executes all **JavaScript code**
- Runs the **event loop**
- Handles **callbacks**
- **Never blocks** (if coded correctly)

### The Helper Threads (Behind the Scenes)
Node.js uses a **thread pool** (libuv) for heavy operations:

// These operations use hidden threads (default 4):
fs.readFile()        // File system operations
crypto.pbkdf2()      // Cryptographic functions
zlib.deflate()       // Compression
DNS.lookup()         // Domain resolution

// Your JavaScript still runs on main thread
// But the waiting happens on other threads

**Important distinction:**
- **JavaScript execution** is single-threaded
- **I/O operations** use multiple threads (or OS async features)
- **You cannot write multi-threaded JavaScript** (without worker threads)

## Real-World Example: 1000 Concurrent Database Queries

// Multi-threaded server (Java)
// 1000 requests = 1000 threads (each blocks on DB query)
ExecutorService executor = Executors.newFixedThreadPool(1000);
for (int i = 0; i < 1000; i++) {
    executor.submit(() -> {
        // Each thread blocks here
        ResultSet result = statement.executeQuery("SELECT * FROM users");
        return result;
    });
}
// Memory: 1000 threads × 1MB = 1GB
// CPU: 1000 threads fighting for CPU time

// Node.js single-threaded server
// 1000 requests = 1 thread (non-blocking queries)
const app = require('express')();

app.get('/users', async (req, res) => {
    // Main thread doesn't block here
    const users = await db.query('SELECT * FROM users');
    res.json(users);
});

app.listen(3000);

// Memory: ~50MB total
// CPU: One thread efficiently handling all requests
// Response time: Same as multi-threaded (or better!)
```

## The Single Thread Advantage

### 1. **No Thread Synchronization Overhead**
// Multi-threaded: Must use locks
synchronized(lock) {
    counter++;  // Expensive
}

// Node.js: No locks needed!
let counter = 0;
app.get('/increment', (req, res) => {
    counter++;  // Safe! Only one thread touches this
    res.json({ counter });
});

### 2. **No Context Switching**

Multi-threaded:
Thread 1 → [work] → context switch (costly) → Thread 2 → [work]
                          ↑
                    1000 threads = 1000s of context switches/second

Node.js:
Thread 1 → [work continuously] → no context switches

### 3. **Lower Memory Usage**

# Multi-threaded server (Java)
$ ps aux | grep java
USER  PID  %MEM  RSS
app   123  45%   4.5GB  # 5000 threads

# Node.js
$ ps aux | grep node
USER  PID  %MEM  RSS  
app   456  2%    128MB  # Single thread, 5000 connections

### 4. **Simpler Programming Model**

// No worry about:
// - Race conditions
// - Deadlocks
// - Thread safety
// - Synchronized collections

// Just write sequential-looking async code
async function handleRequest(req) {
    const user = await db.findUser(req.userId);
    const posts = await db.findPosts(user.id);
    const comments = await db.findComments(posts.map(p => p.id));
    return { user, posts, comments };
}
```

## The Single Thread Disadvantage (Critical!)

### CPU-Intensive Tasks Block Everything

app.get('/compute', (req, res) => {
    // BAD: This blocks the entire server!
    let result = 0;
    for (let i = 0; i < 1e10; i++) {
        result += Math.sqrt(i);
    }
    res.json({ result });
    
    // While this loop runs:
    // - No other requests can be processed
    // - Event loop is frozen
    // - Server appears dead
});

**Demonstration of blocking:**
const express = require('express');
const app = express();

// Route A: Blocks for 5 seconds
app.get('/block', (req, res) => {
    const start = Date.now();
    while (Date.now() - start < 5000) {
        // Blocking loop
    }
    res.send('Done blocking');
});

// Route B: Simple response
app.get('/ping', (req, res) => {
    res.send('pong');
});

// Test:
// 1. curl http://localhost:3000/block
// 2. Immediately curl http://localhost:3000/ping
// Result: /ping waits 5 seconds! (BAD)
```

## Solutions for CPU-Intensive Work

### 1. **Worker Threads** (Node.js 10+)
const { Worker } = require('worker_threads');

app.get('/compute', (req, res) => {
    const worker = new Worker('./compute-worker.js');
    worker.on('message', (result) => {
        res.json({ result });
    });
    worker.postMessage({ data: req.body });
});

// compute-worker.js
const { parentPort } = require('worker_threads');

parentPort.on('message', (task) => {
    // Heavy computation here (doesn't block main thread)
    let result = 0;
    for (let i = 0; i < 1e10; i++) {
        result += Math.sqrt(i);
    }
    parentPort.postMessage(result);
});
```

### 2. **Cluster Module** (Multiple Node.js processes)
const cluster = require('cluster');
const http = require('http');
const numCPUs = require('os').cpus().length;

if (cluster.isMaster) {
    // Fork workers (one per CPU core)
    for (let i = 0; i < numCPUs; i++) {
        cluster.fork();
    }
} else {
    // Each worker has its own single thread
    http.createServer((req, res) => {
        res.writeHead(200);
        res.end('Handled by worker ' + cluster.worker.id);
    }).listen(8000);
}
```

### 3. **Separate Microservices**
```javascript
// API Gateway
app.post('/process-image', async (req, res) => {
    // Offload to dedicated service (Python, Go, etc.)
    const result = await axios.post('http://image-processor:3001/process', req.body);
    res.json(result.data);
});
```

## What the Single Thread Can and Cannot Do

| ✅ **CAN DO** | ❌ **CANNOT DO** |
|--------------|------------------|
| Handle 10,000 concurrent I/O requests | Run CPU-intensive operations |
| Process multiple file reads simultaneously | Execute parallel JavaScript code |
| Handle thousands of WebSocket connections | Utilize multiple CPU cores (without clustering) |
| Wait for hundreds of database queries | Perform true parallel processing |
| Manage timers efficiently | Run blocking loops without freezing |

## The Hidden Thread Pool (libuv)

const crypto = require('crypto');
const fs = require('fs');

// These use the thread pool (default 4 threads)
crypto.pbkdf2('password', 'salt', 100000, 64, 'sha512', (err, hash) => {
    console.log('Crypto done');
});

fs.readFile('large-file.txt', (err, data) => {
    console.log('File read done');
});

// Adjust thread pool size (at startup)
process.env.UV_THREADPOOL_SIZE = 8;

**Thread pool visualization:**
Main Thread (JavaScript)
    │
    ├─> crypto.pbkdf2() ──┐
    ├─> fs.readFile()    ──┼─> Thread Pool (4 threads)
    ├─> dns.lookup()     ──┤
    └─> zlib.deflate()   ──┘
         │
         └─> Callbacks return to Main Thread

## Best Practices for Single-Threaded Architecture

### ✅ **DO:**
// Use async I/O
await fs.promises.readFile('file.txt');

// Use streams for large data
fs.createReadStream('large.csv').pipe(res);

// Use setImmediate to yield
function processLargeArray(items, callback) {
    let index = 0;
    function processChunk() {
        for (let i = 0; i < 1000 && index < items.length; i++) {
            process(items[index++]);
        }
        if (index < items.length) {
            setImmediate(processChunk);  // Yield to event loop
        } else {
            callback();
        }
    }
    processChunk();
}

### ❌ **DON'T:**
// Never use sync I/O
const data = fs.readFileSync('large.txt');  // BLOCKS

// Never do CPU-intensive work
for (let i = 0; i < 1e9; i++) { }  // BLOCKS

// Never block with JSON.parse on huge objects
const huge = JSON.parse(10MBString);  // BLOCKS

## Key Takeaways

| Concept | Explanation |
|---------|-------------|
| **One JavaScript Thread** | All user code executes on single thread |
| **Event Loop** | Manages async operations without blocking |
| **Thread Pool** | 4 threads handle heavy I/O (hidden from you) |
| **No Locks Needed** | No race conditions (simpler code) |
| **Blocking is Fatal** | CPU-intensive code freezes entire server |
| **Scaling** | Use cluster module for multi-core machines |

**The Golden Rule:**
> **Never block the event loop.** Keep your JavaScript thread free to handle events. If you need CPU-intensive work, 
use worker threads or separate services.

**Bottom Line:** Node.js single-threaded architecture is brilliant for I/O-heavy workloads but requires careful 
design to avoid blocking. Think of it as one very efficient waiter who never sleeps, not as a restaurant with many waiters.
