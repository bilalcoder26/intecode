# 1. Functional Requirements

### User should be able to:

* Submit a long URL
* Get a short URL
* Redirect using short URL
* Optional:

  * Custom aliases
  * Expiration time
  * Analytics (click count)

### Example

Long URL:
https://www.example.com/blogs/nodejs-system-design

Short URL:
https://tiny.ly/aB3xYz

# 3. Database Design

### MongoDB Schema

{
    _id: ObjectId,
    shortCode: "aB3xYz",
    longUrl: "https://example.com/blog",
    createdAt: Date,
    expiresAt: Date,
    clicks: 0
}

### Indexes

db.urls.createIndex({ shortCode: 1 }, { unique: true })

db.urls.createIndex({ expiresAt: 1 }, { expireAfterSeconds: 0 })

The first index makes lookups fast.

The second automatically removes expired URLs.

# 5. Generating Unique Short Codes

This is the most important part.

---

## Approach 1: Random String

Generate:

const shortCode = nanoid(6);

Examples:

aB3xYz
K2LmP9

### Pros

* Easy
* Fast

### Cons

* Collision possible

Need uniqueness check:

while(await Url.exists({ shortCode })) {
    shortCode = nanoid(6);
}

---

## Approach 2: Base62 Encoding (Recommended)

Characters:

a-z
A-Z
0-9

Total:

62 characters

### Example

Auto Increment ID:

1 -> b
2 -> c
61 -> 9
62 -> ba

Convert numeric ID to Base62.

123456 -> W7E

### Pros

* No collisions
* Deterministic
* Short URLs

### Cons

* Requires sequence generation

Most interviewers like this approach.

# 6. Node.js Implementation

### Create URL

app.post('/shorten', async (req, res) => {

    const { url } = req.body;

    const shortCode = nanoid(6);

    await Url.create({
        shortCode,
        longUrl: url
    });

    res.json({
        shortUrl: `https://tiny.ly/${shortCode}`
    });
});

---

### Redirect URL

app.get('/:shortCode', async (req, res) => {

    const { shortCode } = req.params;

    const data = await Url.findOne({
        shortCode
    });

    if (!data)
        return res.status(404).send();

    data.clicks++;
    await data.save();

    res.redirect(data.longUrl);
});

---

# 7. Caching with Redis

Without Redis:

Every redirect
→ Database hit

Millions of requests can overload the DB.

---

### Store Mapping

shortCode -> longUrl

Example:

aB3xYz ->
https://example.com/blog

---

### Redirect Flow

Request
   |
Redis Lookup
   |
Hit? ---- Yes ---> Redirect
   |
   No
   |
Database Lookup
   |
Store in Redis
   |
Redirect

Node.js:

const cached = await redis.get(shortCode);

if(cached){
    return res.redirect(cached);
}

This reduces DB load significantly.

# 12. Interview Answer (2-Minute Version)

> I would build the URL shortener using Node.js, Express, Redis, and MongoDB/PostgreSQL. 
When a user submits a long URL, the service generates a unique short code (preferably using Base62 encoding of a sequence ID),
 stores the mapping in the database, and returns a short URL. During redirection, the system first checks Redis for 
 the shortCode→longUrl mapping. On a cache miss, it fetches from the database and populates Redis. 
 For scalability, I'd run multiple Node.js instances behind a load balancer, shard the database by shortCode,
  and use Redis for caching. Analytics events would be sent asynchronously through a queue like RabbitMQ or Kafka to 
  avoid slowing down redirects. This design supports high throughput, low latency, and horizontal scalability.

This is typically the level of detail expected for a senior Node.js system design interview question.
